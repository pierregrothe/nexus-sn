# Story 2026.05.02: CLI adaptive batch progress with weighted ETA

Status: backlog
Spec-Clarity: high
Depends-On: 2026.05.01

## Story

As a NEXUS operator running `nexus plugins upgrade <id>` or
`nexus plugins updates --apply` on a ServiceNow instance,
I want NEXUS to pick the right progress display for my terminal
automatically -- a rich Live region with brand bars and weighted
ETA on a modern TTY, a plain "12/17 ETA 03:42" status line on
legacy cmd.exe or piped output -- so that long batch operations
are visible without staring at silent logs.

## Acceptance Criteria

AC1:
**Given** a RICH or BASIC profile and a `nexus plugins upgrade
<id>` invocation
**When** the operation polls SN's progress endpoint
**Then** a per-item Rich progress bar renders with: brand-styled
spinner (RICH only), plugin id, current percent (from SN),
elapsed time, and `WeightedETAColumn` showing `ETA: estimating...`
on first tick and `ETA: MM:SS` once duration data is collected.

AC2:
**Given** a RICH or BASIC profile and a `nexus plugins updates
--apply --yes` invocation with N >= 2 pending plugins
**When** the batch starts
**Then** an overall Rich progress bar renders with `M of N`,
elapsed batch time, weighted batch ETA -- plus a per-item
transient task that displays the in-flight plugin's progress and
vanishes on item completion.

AC3:
**Given** a LEGACY or PLAIN profile and any `nexus plugins
upgrade <id>` invocation
**When** the operation polls
**Then** a single line `upgrading <plugin_id> <sn_pct>%` is
printed on each SN poll cycle (no Live region, no `\r` rewrites,
multiplexer-safe). Final line `upgraded <plugin_id> MM:SS` on
success or `failed <plugin_id> MM:SS <reason>` on failure.

AC4:
**Given** a LEGACY or PLAIN profile and a `nexus plugins updates
--apply` invocation
**When** the batch runs
**Then** one completion line per item plus a final summary
`batch complete: M of N succeeded, batch elapsed MM:SS, batch
ETA was MM:SS final`. No live updates, no progress bars.

AC5:
**Given** any batch on item 1
**When** no completed-item duration data exists yet
**Then** the batch ETA shows `ETA: estimating...` (literal,
dim theme token in RICH/BASIC; plain text in LEGACY/PLAIN).

AC6:
**Given** a batch on item k where k >= 2 and at least one prior
item completed
**When** the ETA renders
**Then** it shows `ETA: MM:SS` computed as `remaining_full_items
* ema_duration + (1 - sn_pct) * ema_duration` with alpha=0.4.

AC7:
**Given** `EmaPriorStore.record(family, duration_s)` is called
**When** the call completes
**Then** a JSON line `{"family": "...", "duration_s": F, "ts":
"2026-...T...Z"}` is appended to
`~/.nexus/cache/eta_prior.jsonl` (UTC timestamp).

AC8:
**Given** two concurrent `nexus plugins updates --apply`
invocations on the same machine
**When** both write to `EmaPriorStore` simultaneously
**Then** all records from both invocations are preserved.

AC9:
**Given** `EmaPriorStore.load(family)` is called and the JSONL
file contains more than 1000 entries
**When** the loader runs
**Then** it returns the 1000 most recent entries for the
requested family; file is unchanged.

AC10:
**Given** `PluginExecutor.batch_upgrade(targets, families=(),
console=console, progress=None)`
**When** progress is None
**Then** behavior is preserved exactly as today (skip-on-fail
sequential loop, per-plugin `console.print` lines).

AC11:
**Given** `PluginExecutor.batch_upgrade(..., progress=
some_batch_progress)`
**When** the loop executes
**Then** all output that today goes to `console.print` flips to
`some_batch_progress.console.print` so it interleaves with the
Live region in RICH/BASIC; in LEGACY/PLAIN, the same path emits
plain lines.

AC12:
**Given** `make_batch_progress(ctx, total=N)` is called
**When** `ctx.profile` is RICH or BASIC
**Then** the factory returns a `RichBatchProgress` instance.
**When** `ctx.profile` is LEGACY or PLAIN
**Then** the factory returns a `PlainBatchProgress` instance.
**Then** both instances satisfy `BatchProgressProtocol` (same
method surface).

AC13:
**Given** the test suite
**When** tests run
**Then** they use a `FakeBatchProgress` (records start_item /
update_item / finish_item calls) for executor-layer tests, and
`Console(force_terminal=True/False, file=StringIO())` for
RichBatchProgress / PlainBatchProgress unit tests. No mocks of
Rich internals.

AC14:
**Given** `nexus plugins updates --apply` invoked under PLAIN
profile WITHOUT `--yes`
**When** the command runs
**Then** it raises `InteractiveRequiredError` with exit code 3
(confirmation prompt is suppressed under PLAIN; `--yes` is
required for non-interactive use).

AC15:
**Given** `nexus plugins updates --apply --yes` invoked under
PLAIN profile
**When** the command runs
**Then** the batch proceeds with `PlainBatchProgress` output;
no Live region, no `\r`, suitable for CI logs.

## Must NOT

- Must NOT hard-code per-family ETA priors (rejected in
  brainstorm round 1).
- Must NOT use a JSON dict / single-blob cache file for ETA data
  (race-condition prone). JSONL only.
- Must NOT call `console.print(...)` directly inside
  `batch_upgrade` when a progress is provided. All output goes
  through `progress.console.print`.
- Must NOT change `PluginExecutor.upgrade`'s OR
  `PluginExecutor.batch_upgrade`'s existing positional argument
  order. Add `progress` as a keyword-only argument with default
  None.
- Must NOT depend on `tqdm` or any progress library other than
  Rich.
- Must NOT use `\r` carriage-return rewrites in
  `PlainBatchProgress` (corrupts output in tmux / screen /
  certain CI consoles). One full line per event, period.
- Must NOT add asyncio.Lock around `EmaPriorStore` writes
  (append-only file mode is sufficient).
- Must NOT block on missing prior data. The store returns an
  empty list on first read for an unknown family.
- Must NOT depend on `BatchProgress` having a no-op fallback
  inside the class. The factory picks the right implementation
  based on profile; each implementation is honest about what it
  does.

### Testing Approach

- Real fakes only. `FakeBatchProgress` records `start_item /
  update_item / finish_item` calls. `Console(force_terminal=
  True/False, file=StringIO())` for routing tests.
- `EmaPriorStore` concurrent-write tests use real threading +
  real tempfile.
- `WeightedETAColumn.render` is unit-testable directly -- pass a
  hand-built `rich.progress.Task` with the expected fields.
- `RichBatchProgress.__enter__` / `__exit__` tested via Rich's
  standard Live-test pattern (capture stdout, assert no leak on
  context exit).
- Coverage gate: 100% line for both new files. ratchet.json
  updated.

### Conventions

- File headers on every new `.py` file.
- Pydantic frozen + strict + extra="forbid".
- Google-style docstrings on every public symbol.
- `__all__` on every module.
- Absolute imports only.
- UTC for all timestamps in `EmaPriorStore` (`datetime.now(UTC)`
  via `from datetime import UTC`).
- No `# type: ignore`.
- ASCII-only.
- Test naming: `test_<function>_<scenario>`.
